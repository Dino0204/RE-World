## 동작 방식

1. `git diff main..HEAD` 또는 최근 커밋을 확인하여 변경 사항을 파악한다
2. 변경된 파일의 코드를 읽고 작업 맥락을 이해한다
3. 사용자에게 추가 맥락이 필요하면 질문한다
4. 작업 내용이 버그 수정이면 "버그 이슈" 템플릿, 그 외에는 "새 이슈" 템플릿을 사용한다
5. 요약을 작성하여 Notion 보드에 페이지를 생성한다

## 속성 매핑

페이지 생성 시 다음 속성을 설정한다:

- 이름: [ MCP ] 작업 내용을 한 줄로 요약
- 이슈 타입: "버그" / "새 기능" / "개선사항" 중 적절한 것
- 상태: 작업이 끝났으면 "완료", 진행 중이면 "진행 중"
- 우선순위: 사용자에게 확인 (기본값 "중간")
- date:시작일자:start: 작업 시작일 (사용자에게 확인)
- date:완료일자:start: 오늘 날짜 (완료 시)
- Test Environment: 해당되는 경우에만 설정

## 페이지 본문 구조

```markdown
## 작업 개요

한 줄 요약

## 문제 상황

어떤 문제가 있었는지 (증상, 에러 메시지 포함)

## 해결 과정

시도한 접근법과 결과를 시간순으로 기록

- [AI 제안] 또는 [직접 판단] 태그 표기
- 구분이 불명확하면 사용자에게 확인

## 최종 해결책

실제 적용된 방법과 그 이유

## 주의사항 / 후속 작업

알려진 제한사항, TODO, 향후 개선점
```
